# ナレッジ抽出・再利用 設計メモ

Phase 5 以降のコア部分。研究的要素が大きいので、ここでは「何を作りたいか」「どういう問いがあるか」「考えられるアプローチ」を整理する。実装の詳細は走らせながら決める。

## 動機

Claude Code を継続的に使っていると、以下のようなものが履歴に蓄積される:

- 過去に解決したバグと、そのアプローチ
- 試して上手くいかなかった方法
- プロジェクト固有の慣習 (このリポジトリでは○○というディレクトリ命名)
- よく使うコマンド・スニペット
- 失敗パターン (このライブラリのこの呼び出し方は罠)
- レビューで指摘された規約

これは**そのプロジェクト・チーム・個人の経験知**だが、JSONL に埋もれたままでは活用できない。新しいセッションを始めるたびに同じ説明を Claude にしているのは無駄。

**目標**: 履歴を「読める形式」に蒸留し、新しいセッションの開始時に活用できるようにする。

## 用語の整理

- **セッション**: 1 つの `.jsonl` ファイル。1 回の Claude Code 起動からコンパクションを挟んでも論理的に繋がっている範囲
- **アノテーション**: 人間が後付けで足す情報 (タグ・メモ・お気に入り)。Phase 4
- **サマリ**: セッション単位の自動要約。Phase 5
- **ナレッジアイテム** (Knowledge Item): 履歴から抽出した再利用可能な知見の単位。Phase 5
- **AGENT.md**: ナレッジアイテムから組み立てた、新セッションに渡すコンテキスト。Phase 6

## ナレッジアイテムの種類 (案)

`knowledge_items.type` の取りうる値:

| type         | 意味                 | 例                                              |
| ------------ | -------------------- | ----------------------------------------------- |
| `pattern`    | 上手くいくやり方     | 「この種のテストは pytest fixture で書く」      |
| `pitfall`    | 罠 / 失敗パターン    | 「`async with` の中で `await` を忘れがち」      |
| `convention` | プロジェクト規約     | 「コミットメッセージは Conventional Commits」   |
| `snippet`    | 再利用可能なコード片 | 「DB 接続のセットアップコード」                 |
| `decision`   | 過去の意思決定       | 「ORM は SQLAlchemy ではなく Drizzle を選んだ」 |
| `tool-usage` | ツール使い方の知見   | 「Grep で N 行コンテキストを出すには `-C N`」   |

スコープ (`scope` フィールド):

- `user`: 個人の知見 (このユーザーがどこに行っても役立つ)
- `project`: プロジェクト固有 (このリポジトリでだけ役立つ)
- `org`: 組織全体 (全員で共有したい)

## 主要な研究的問い

### Q1: 抽出の粒度はどうあるべきか

- セッション単位の要約だけで十分か?
- メッセージ単位の細かい知見も拾うべきか?
- ターン単位 (user → assistant のペア) はどうか?

**仮説**: 粒度を 3 階層持つ。

1. メッセージ embedding (検索用)
2. セッションサマリ (一覧・類似度用)
3. ナレッジアイテム (人が読む形)

下から上に蒸留していく。

### Q2: 何を「ナレッジ」として残すべきか

履歴の大部分はノイズ。「Claude Code が考えながらツールを使った内部活動」と「再利用できる知見」を区別する必要がある。

**仮説**: 以下のシグナルを持つメッセージを優先的に拾う:

- ユーザーが明示的に修正・訂正した箇所 (= 学習機会)
- 同じパターンが複数セッションに現れる
- 長いセッションの中で「決定的瞬間」(問題が解決した瞬間)
- レビューコメント的なやり取り

### Q3: 抽出方法

候補:

- **LLM プロンプト**: セッション全体を投げて「再利用できる知見を抽出して」と頼む
  - 単純だが、長いセッションでは context window 問題
  - 階層的要約 (map-reduce) で回避可能
- **ルールベース + LLM**: コードブロック・コマンド・ファイル変更を機械的に抽出 → LLM で説明文を付ける
- **横断分析**: 複数セッションを横断して「繰り返し現れるパターン」を統計的に発見

**仮説**: 単一セッション抽出 (LLM) → 横断クラスタリング (embedding) → LLM で命名・整理、のパイプライン。

### Q4: 信頼性をどう担保するか

LLM が抽出したものは幻覚を含む可能性がある。

**原則**:

- 全てのナレッジアイテムは `source_sessions` で出典を保持
- `status` は `draft` から始まり、人間がレビューして `reviewed` に昇格
- AGENT.md には `reviewed` のものだけを使う (オプションで `draft` も含められる)
- 出典セッションに 1 クリックで飛べる UI

### Q5: 個人知 vs プロジェクト知 vs 組織知の境界

- プロジェクトの規約は「全員で共有したい」
- でも個人の作業メモは見られたくないかも
- 組織全体の知見はチームで合意したい

**仮説**: 3 つのスコープを最初から分けておく。デフォルトは `user` スコープ、明示的に `project` や `org` に昇格させる UI を用意。

### Q6: AGENT.md をどう組み立てるか

候補:

- 全ナレッジアイテムを単純連結 → 長すぎる
- LLM で「このプロジェクトに関連する知見をまとめて」 → 動的だが再現性が低い
- テンプレート + 構造化データ → 安定するが柔軟性が低い

**仮説**: テンプレート + 構造化データを基本にしつつ、各セクションは編集可能。新しいセッションを取り込んだ時の差分提案 UI を別に作る。

## アーキテクチャ案

```
JSONL (NAS)
   │
   ▼
[セッションパーサー]  ← Phase 1
   │
   ▼
[メッセージ DB]   [embedding DB]
   │                    │
   ▼                    ▼
[セッションサマリ生成 (LLM)]   [類似度検索]
   │
   ▼
[ナレッジ抽出パイプライン (LLM)]
   │
   ▼
[knowledge_items テーブル]
   │
   ├──→ [ナレッジ管理 UI] (CRUD・スコープ・タグ・承認)
   │
   └──→ [AGENT.md 生成器]  ← Phase 6
              │
              ▼
        [MCP サーバー] ← Phase 7
              │
              ▼
        新しい Claude Code セッション
```

## ナレッジ管理アプリの UI 構想

別タブとして以下の画面を持つ:

### `/knowledge` — ナレッジ一覧

- フィルタ: type, scope, status, tag
- 各アイテムをカード表示
- 出典セッション数のバッジ
- 編集・削除・スコープ変更

### `/knowledge/{id}` — ナレッジ詳細

- title / body の編集
- type / scope の変更
- 出典セッションへのリンク (Phase 1〜3 の履歴ビューアに飛ぶ)
- 「関連ナレッジ」(embedding で類似のものを提示)

### `/knowledge/extract` — 抽出パイプライン管理

- 「未抽出のセッション」一覧
- 「これらから抽出」ボタン (LLM ジョブ起動)
- 抽出中のジョブ進捗
- 結果を `draft` として保存

### `/projects/{user}/{project}/agent-md` — AGENT.md プレビュー

- このプロジェクトに紐づくナレッジから生成された Markdown のプレビュー
- セクション単位の編集
- 「ダウンロード」「クリップボードにコピー」「リポジトリに書き出す」
- 「最新セッションを取り込む」差分提案

## 評価指標 (Phase 5 完了基準)

定性的だが以下を満たしていれば成功:

- [ ] 過去 1 ヶ月のセッションから少なくとも 10 個の「使える」ナレッジが抽出されている
- [ ] 抽出されたものを読んだ実際のユーザーが「これは確かに自分のやり方だ」と認識する
- [ ] 出典セッションに飛んで根拠を確認できる
- [ ] 幻覚率 (出典に書かれていないことが書かれている割合) が許容範囲

## 評価指標 (Phase 6 完了基準)

- [ ] 生成された AGENT.md を新しい Claude Code セッションに与えると、明らかに有用な改善 (規約遵守・既知の落とし穴回避) が見られる
- [ ] AGENT.md が長すぎず短すぎない (context window を圧迫しない)
- [ ] 古くなった内容 (もう使っていないライブラリの話など) が自動で減衰または除外される仕組みがある

## 未解決の論点

- **古い知見の扱い**: プロジェクトの方針転換でかつてのベストプラクティスが嘘になる。embedding だけでは古さを判定できない。LLM に「最近のセッションで矛盾するパターンが出ているか」を聞く?
- **個人の癖 vs プロジェクトの規約**: 個人のスタイル (例: 変数名の好み) と全員で守るべき規約をどう区別するか
- **複数ユーザーの知見の衝突**: A さんの「こうやる」と B さんの「こうやる」が違う時にどう統合するか
- **コスト**: 全セッションを LLM で処理するとそれなりにかかる。差分処理・サンプリングなどの工夫が必要
